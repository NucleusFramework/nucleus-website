## Output rules
- Every page is BILINGUAL: write `name.mdx` (English, default) AND `name.fr.mdx` (French) side-by-side, same folder, same `meta.json` order.
- Frontmatter on every page:
  ```yaml
  ---
  title: <Value-led, concrete>
  description: <One sentence: what does this give me?>
  ---
  ```
- Page beats — apply consistently:
  1. **Lede** (2-3 sentences, no heading) — problem + what this solves
  2. `## TL;DR` — 3-5 scannable bullets
  3. `## Install` — Gradle coordinate from inventory, or "included in `nucleus-application`"
  4. `## Quickstart` — minimal working snippet
  5. `## How it works` — 2-3 paragraphs of mental model
  6. `## Reference` — API surface organized by use case
  7. `## Notes / Troubleshooting` — only if real gotchas exist

## Voice
- **Marketing-tinted but factual**. Open with the "why care?" angle. Lift phrasings from the landing where they fit ("Cold start of a binary, throughput of the JVM", "One language, every screen", etc.)
- **Senior dev energy**. No "let's start by..." or "now we'll explore...". State things.
- **Show first, explain second**. Every concept gets a snippet within the first screen.
- **Active voice**. "Nucleus exposes X" not "X is exposed by Nucleus".
- **No emojis** in body.
- **English first** — write EN page first, then translate to natural French. Don't word-by-word translate; reshape sentences.

## French notes
- Use **vouvoiement** ("vous") for the dev.
- Keep code identifiers in English (function names, class names — those don't translate).
- Translate UI strings, descriptions, headings.
- File names: append `.fr` before `.mdx`: `quickstart.mdx` → `quickstart.fr.mdx`.

## Code snippets
- Kotlin: assume Kotlin 2.x, Compose Multiplatform, JDK 17+.
- Gradle: `build.gradle.kts` Kotlin DSL, with `id("dev.nucleusframework")` plugin.
- Always show the IMPORT lines if non-obvious. Most types live under `dev.nucleusframework.<module>.*`.
- Use placeholder values when needed: `com.example.MyApp`, `myapp`, etc.

## Fumadocs MDX components available
- `<Callout type="info|warn|error|tip">` — for admonitions
- `<Tabs items={['A', 'B']}><Tab value="A">…</Tab><Tab value="B">…</Tab></Tabs>` — for tabbed content
- Standard markdown for everything else
- If you import components, add the import line right after the closing `---` of frontmatter.

## Cross-links
- Use absolute `/docs/...` paths (the EN version). Fumadocs i18n auto-resolves to current language.
  Example: `[the Tao backend](/docs/tao)`
- ALWAYS link to related pages in the "How it works" or "Notes" section.

## Anti-patterns to avoid
- Long preamble before showing code
- Restating mkdocs syntax (`!!! note` etc — use Callout)
- Pretending APIs exist that aren't in the inventory
- Adding emojis
- "Welcome to the docs!" filler
- Translating identifier names into French
