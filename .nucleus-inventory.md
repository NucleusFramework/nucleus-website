# Nucleus 2.0 source inventory

Maven group: `dev.nucleusframework` for runtime libraries. Artifact IDs follow `nucleus.<module-name>`. The published version is derived from the `GITHUB_REF` tag (`refs/tags/vX.Y.Z`), defaulting to `1.0.0` in local builds — when the README/website calls the line "Nucleus 2.0" it refers to the major release, but the on-disk `coordinates(...)` calls use `publishVersion` (CI-injected).

Source: `~/IdeaProjects/Nucleus/`, root package `dev.nucleusframework`. All modules are JVM-only Kotlin (no KMP `commonMain`); platform fan-out is done via runtime detection inside a single `src/main/kotlin` tree.

Notes / stubs:
- `clipboard-common`, `clipboard-macos`, `webview-macos`, `linux-hidpi` mentioned in the inventory request — only `linux-hidpi` is wired into `settings.gradle.kts`. `clipboard-*` and `webview-macos` have **no Kotlin sources** in tree (`src/main/kotlin` empty / missing) and are not in `settings.gradle.kts` — they are placeholders, omitted below.
- `aot-runtime` is mostly a typealias forwarder to `core-runtime`.

## aot-runtime

- **Deps**: `core-runtime`

## autolaunch
- **Artifact**: `dev.nucleusframework:nucleus.autolaunch`
- **Purpose**: Cross-platform "launch at login" — registers the app to start at user login. Windows auto-detects MSIX vs Win32, macOS uses `SMAppService.mainApp` (macOS 13+), Linux uses systemd-user units or the Flatpak portal.
- **Platforms**: macOS, Windows, Linux
- **Public API**:
  - `enum class AutoLaunchState { ENABLED, DISABLED, DISABLED_BY_USER, UNSUPPORTED, ... }`
  - `enum class AutoLaunchResult`
  - `object AutoLaunchConfig` — DSL for backend-specific config
  - `object AutoLaunch { fun preload(); fun state(): AutoLaunchState; fun enable(): AutoLaunchResult; fun disable(): AutoLaunchResult; fun openSystemSettings(); fun wasStartedAtLogin(args: Array<String>): Boolean; fun diagnostic(): String }`
- **Deps**: `core-runtime`. Per-platform JNI bridges (`Win32RegistryBackend`, `MsixStartupTaskBackend`, `MacSMAppServiceBackend`, `SystemdUserBackend`, `FlatpakPortalBackend`).
- **Usage**:
  ```kotlin
  when (AutoLaunch.state()) {
      AutoLaunchState.ENABLED -> {}
      AutoLaunchState.DISABLED -> AutoLaunch.enable()
      AutoLaunchState.DISABLED_BY_USER -> AutoLaunch.openSystemSettings()
      else -> {}
  }
  ```

## Cross-module wiring

- **`core-runtime`** is the dependency root. It provides `Platform.Current`, `ExecutableRuntime.type()`, `NucleusApp.appId/version/aumid/...`, the `SingleInstanceManager`, `DeepLinkHandler`, and `NativeLibraryLoader`. Every other runtime module depends on it.
- **The Gradle plugin (`plugin-build`)** injects a `nucleus/nucleus-app.properties` classpath resource into the packaged app containing `app.id`, `app.version`, `app.vendor`, `app.aumid`, `startup.task.id`, etc., which `NucleusApp` reads at runtime. This is how DSL-level config flows to library code at runtime.
- **`aot-runtime`** just re-exports `core-runtime.ExecutableRuntime` and adds `AotRuntime.mode()` reading the `nucleus.aot.mode` system property — the plugin sets it before launching the JVM in training mode.
- **`graalvm-runtime`** is initialized from `nucleus-application` (`GraalVmInitializer.initialize()` is called inside `nucleusApplication { ... }`). Its SVM substitutions are baked into the native image at build time when the GraalVM build pipeline runs.
- **`nucleus-application`** is the user-facing entry point. It:
  1. Calls `GraalVmInitializer.initialize()`.
  2. Optionally acquires the single-instance lock.
  3. Primes platform integrations (deep links, etc.).
  4. Resolves the backend (probe `dev.nucleusframework.window.tao.TaoApplication` on the classpath → `Tao`; else `Awt`).
  5. Calls `application { … }` (Compose Desktop) or `taoApplication { … }` (from `decorated-window-tao`).
- **`decorated-window-core`** holds Compose styling shared across `-awt`, `-jbr`, `-jni`, `-tao`, `-jewel`, `-material2`, `-material3`. The five style modules add `@Composable` overloads on `DecoratedWindowScope` etc., the four backends (`-jbr` / `-jni` / `-awt` (intermediate) / `-tao`) provide the concrete `DecoratedWindow` Composables.
- **`-jbr` and `-jni`** both depend on `decorated-window-awt`, which holds the AWT-side Compose code. Only their per-OS bridge differs (JBR custom decorations vs raw JNI on stock JDK).
- **`-tao`** is the no-AWT path; it brings its own `ApplicationScope`, `TaoWindow`, native dispatcher (`TaoMainDispatcher`), drag-and-drop bridge, deep-link bridge, native-view embedding, popup hosting and Metal/GL Skiko renderer.
- **`nucleus-application`** then layers `NucleusWindow` (backend-agnostic handle), `NucleusDecoratedWindowScope`, and `LocalNucleusWindow` on top so user code can stay portable across backends. `nucleusWindow.unsafe.taoWindow` / `unsafe.awtWindow` are the escape hatches.
- **Toolkit modules** (`autolaunch`, `notification-*`, `media-control`, `global-hotkey`, `energy-manager`, `system-info`, `system-color`, `taskbar-progress*`, `launcher-*`, `menu-macos`, `scheduler`, `service-management-macos`, `updater-runtime`, `native-ssl`, `native-http*`, `darkmode-detector`, `linux-hidpi`, `sf-symbols`, `freedesktop-icons`) are independent leaves — each depends only on `core-runtime` (some on `system-info` or `notification-common`) and is opt-in.
- **`taskbar-progress-tao`** specifically depends on `nucleus-application` because it adds `NucleusWindow.setTaskbarProgress(...)` extension functions.
- **Native libraries**: every JNI module ships per-OS Rust/C sources under `src/main/native/{macos,linux,windows}/`. The root build registers a `buildNative` task that fans out to platform-specific `buildNativeMac` / `buildNativeWindows` / `buildNativeLinux` tasks (only the current host's is enabled outside CI), and packaged shared libs go into `src/main/resources/nucleus/native/…` to be extracted by `NativeLibraryLoader` at runtime.
