# Nucleus docs — writing style guide (v2)

This is the **binding** style for every page under `content/docs/`. It replaces the
marketing-tinted voice described in `.writing-brief.md`. When the two conflict, this file wins.

The target register is the JetBrains **Kotlin Multiplatform docs**, specifically
`topics/quickstart.md` (cloned locally at `/tmp/kmp-dev-docs/topics/`). Read that file
before writing. The goal is that a reader can't tell our docs and the KMP docs apart.

---

## 1. Voice

- **Second person, present tense.** Address the reader as "you". Use the imperative for
  instructions: "Add the dependency", "Run the app", not "We will add" or "The user should add".
- **French pages use tutoiement** ("tu"), never "vous".
- **Neutral and factual.** State requirements and behavior plainly. No hedging
  ("you'll probably want", "it may be possible"). KMP writes: *"The plugins necessary for
  Kotlin Multiplatform require at least IntelliJ IDEA 2025.2.2."* — copy that flatness.
- **No marketing.** Delete hype adjectives and sales framing. Banned words/phrases:
  *blazing, first-class, unlocks, productized, rounds out, real desktop framework,
  peak throughput, instant, store-ready, à la carte, powerful, seamless, effortless,
  cutting-edge, best-in-class*. Say what the thing does, not why it's great.
- **No "why care?" ledes.** Do not open a page selling the reader on the feature.
  Open by stating what the page covers or what the API does.
- **Explain, don't sell.** A comparison table is fine when it's factual; superlatives are not.

## 2. Page opening

- **No H1** — the frontmatter `title` renders the heading. The first line of the body is the lede.
- **Lede = 1–2 plain sentences.** State what the page covers or what the module does.
  - Tutorial/procedural page: `In this tutorial, you'll ...` / `On this page, you'll ...`
  - Reference/module page: `<Module> lets you <do X> from Kotlin.` Then one sentence on scope.
- **Kill `## TL;DR` entirely.** The KMP docs have no TL;DR blocks. Remove them everywhere.

## 3. Headings

- **Task-based imperative** for anything procedural: `Set up the project`, `Add the dependency`,
  `Send a notification`, `Run the app`, `Package the app`, `Sign the build`.
- Reference sections may use noun headings (`API reference`), but the sub-headings under them
  should still be task-oriented (`Build a notification`, `Send`, `Check availability`).
- Sentence case, not Title Case: `Run the app`, not `Run The App`.
- Keep the closing section titled exactly **`## What's next`** (FR: `## Et ensuite`).

## 4. Structure per archetype

Full conversion applies to every page. Use the skeleton that matches the page's job.

**A. Getting-started / tutorial** (`start/*`, `migrate/*`, packaging how-tos, CI/CD):
```
<lede: "In this tutorial, you'll ...">
## Before you start          (prerequisites, as a bullet list — optional)
## <Step 1 imperative>       (numbered steps inside, code per step)
## <Step 2 imperative>
...
## What's next
```

**B. Reference / module page** (`os/*`, `lifecycle/*`, `performance/*`, `ecosystem/*`):
```
<lede: "<Module> lets you ... from Kotlin.">
## Add the dependency        (Gradle coordinate, verified against source)
## <Primary task>            (minimal working snippet first, then prose)
## How it works              (2–3 short paragraphs — mental model, no hype)
## API reference             (organized by task, not by class)
## Notes                     (only if real gotchas — else omit)
## What's next
```

**C. Concept page** (`concepts/*`, section `index.mdx`, `tao/index.mdx`):
```
<lede: plain statement of the concept>
## <named subsections>       (noun or task headings, whichever reads plainer)
## What's next
```

- The site `index.mdx` and `why-nucleus.mdx` are the only overview pages; keep them
  informative but still strip hype. They may keep a short feature list (no `## TL;DR` label).

## 5. Admonitions (Writerside → Fumadocs)

The KMP docs use `> ... {style="note"}`. We render with Fumadocs `<Callout>`:

| KMP / intent            | Fumadocs                          |
|-------------------------|-----------------------------------|
| `{style="note"}`, info  | `<Callout type="info">`           |
| tip                     | `<Callout type="tip">`            |
| warning / caution       | `<Callout type="warn">`           |
| error / danger          | `<Callout type="error">`          |

- Keep callouts **short and single-idea**, like KMP. One or two sentences.
- Add the import once, right after frontmatter:
  `import { Callout } from 'fumadocs-ui/components/callout';`

## 6. Tabs (platform / tool variants)

KMP uses `<tabs><tab>` heavily for platform-specific paths. We use:
```mdx
import { Tab, Tabs } from 'fumadocs-ui/components/tabs';

<Tabs items={['macOS', 'Windows', 'Linux']}>
<Tab value="macOS"> ... </Tab>
<Tab value="Windows"> ... </Tab>
<Tab value="Linux"> ... </Tab>
</Tabs>
```
Use tabs for genuinely divergent per-OS or per-tool instructions, not to hide length.

## 7. Code

- Kotlin 2.x, Compose Multiplatform, JDK 17+. Gradle: `build.gradle.kts` (Kotlin DSL).
- **Show imports** when the type isn't obvious. Runtime types live under `dev.nucleusframework.<module>.*`.
- Add a filename title to build snippets: ```` ```kotlin title="build.gradle.kts" ````.
- Shell blocks use `shell` (matches KMP), e.g. ```` ```shell\n./gradlew run\n``` ````.
- **Verified coordinates only.** The Maven group is `dev.nucleusframework`; artifacts are
  `nucleus.<module-name>`. Use version `2.0.0` in tutorials, `<version>` is acceptable in reference.
  Never document an artifact that is not in the source's `settings.gradle.kts`.
- Placeholders: `com.example.MyApp`, `myapp`.

## 8. Fact-checking protocol (non-negotiable)

Before writing any API, verify it against the real source at
`/home/elie-gambache/IdeaProjects/Nucleus/<module>/src/main/kotlin/...`.

1. Trust `.nucleus-inventory.md` for the shape, but it may be **stale** — the source is truth.
2. If the existing page references a type, function, or artifact that is **not** in the source,
   correct it or remove it. Note the removal in your summary.
3. Never invent APIs. If unsure a symbol exists, grep for it before documenting it.
4. Match signatures exactly: enum values, function names, DSL property names.

## 9. Cross-links

- Absolute `/docs/...` paths to the **English** slug; Fumadocs i18n resolves to the current
  language. Example: `[the Tao backend](/docs/tao)`.
- Every page ends with `## What's next` linking 2–4 related pages.

## 10. French pages

- Same file, `.fr.mdx` suffix, same folder, same `meta.json` order.
- Tutoiement. Translate prose, headings, callout titles/bodies, and descriptions.
- **Do not translate identifiers** — class, function, property, enum, artifact names stay English.
- Translate headings to imperative French: `Add the dependency` → `Ajouter la dépendance`,
  `Run the app` → `Lancer l'application`, `Package the app` → `Empaqueter l'application`,
  `How it works` → `Fonctionnement`, `Notes` → `Notes`, `What's next` → `Et ensuite`,
  `Before you start` → `Avant de commencer`, `API reference` → `Référence de l'API`.
- Reshape sentences to read naturally in French — never word-for-word translate.

## 11. Anti-patterns (reject on sight)

- `## TL;DR` blocks.
- Marketing ledes / hype adjectives (see §1 ban list).
- Long preamble before the first code snippet on a procedural page.
- Documenting APIs not present in the source.
- Emojis in body text.
- "Welcome to the docs!" filler.
- Title Case headings.
- Translating identifier names into French.

## Worked example — `start/quickstart.mdx`

This is the gold standard. Every conversion should feel like this.

````mdx
---
title: Quickstart
description: Build and run your first Nucleus desktop app, then package it as a native installer.
---

import { Callout } from 'fumadocs-ui/components/callout';

In this tutorial, you'll create a Nucleus desktop app from scratch, run it as a native
Tao window, and package it into an installer for your current operating system. The whole
app is two files: a Gradle build script and a Kotlin `main` function.

## Before you start

Nucleus builds on Compose Multiplatform and requires:

- **JDK 17 or later** on your `PATH`. [JetBrains Runtime](https://github.com/JetBrains/JetBrainsRuntime)
  is recommended when you use the AWT backend.
- **Kotlin 2.0 or later** and **Gradle 8.x**.

## Create the build script

In an empty project directory, create `build.gradle.kts` and apply the Nucleus plugin:

```kotlin title="build.gradle.kts"
plugins {
    kotlin("jvm") version "2.0.21"
    id("org.jetbrains.compose") version "1.7.0"
    id("dev.nucleusframework") version "2.0.0"
}

repositories {
    mavenCentral()
    google()
}

dependencies {
    implementation(compose.desktop.currentOs)
    // Enable the Tao backend (Rust-native windowing)
    implementation("dev.nucleusframework:nucleus.decorated-window-tao:2.0.0")
}

nucleus.application {
    mainClass = "com.example.MainKt"

    nativeDistributions {
        packageName = "MyApp"
        packageVersion = "1.0.0"
        targetFormats(TargetFormat.Dmg, TargetFormat.Msi, TargetFormat.Deb)
    }
}
```

## Write the application

Create `src/main/kotlin/com/example/Main.kt`:

```kotlin
package com.example

import androidx.compose.foundation.layout.Box
import androidx.compose.foundation.layout.fillMaxSize
import androidx.compose.material.Text
import androidx.compose.ui.Modifier
import dev.nucleusframework.nucleusapplication.DecoratedWindow
import dev.nucleusframework.nucleusapplication.NucleusBackend
import dev.nucleusframework.nucleusapplication.nucleusApplication

fun main() = nucleusApplication(backend = NucleusBackend.Tao) {
    DecoratedWindow(
        onCloseRequest = ::exitApplication,
        title = "MyApp",
    ) {
        Box(Modifier.fillMaxSize()) {
            Text("Hello from Nucleus")
        }
    }
}
```

The `nucleusApplication` entry point starts the Nucleus runtime and opens a `DecoratedWindow` —
a Compose window with a native title bar.

<Callout type="info">
`NucleusBackend.Auto` (the default) picks Tao when `decorated-window-tao` is on the classpath,
and falls back to the AWT backend otherwise. Force a backend with `NucleusBackend.Tao` or
`NucleusBackend.Awt`. See [Backends](/docs/concepts/backends).
</Callout>

## Run the app

```shell
./gradlew run
```

A native window opens with a custom title bar — drawn by Tao on Linux and Windows, and by
the system on macOS.

## Package the app

```shell
./gradlew packageDistributionForCurrentOS
```

The installer is written to `build/compose/binaries/main/<format>/`: a `.dmg` on macOS, an
`.msi` on Windows, and a `.deb` on Linux. Add formats by extending `targetFormats(...)` —
Nucleus supports 16. See [Configuration](/docs/start/configuration).

## What's next

- [Project setup](/docs/start/project-setup) — multi-module layout, Gradle properties, and JDK config.
- [Configuration](/docs/start/configuration) — the full `nucleus { }` DSL.
- [Architecture](/docs/concepts/architecture) — what runs under your `main` function.
````

Note what changed from the original: the marketing lede and `## TL;DR` are gone; headings are
task-imperative; shell blocks use `shell`; the closer is `## What's next`. The facts (APIs,
coordinates, tasks) were preserved — but still verify them against the source per §8.
