## v2.0.0

**Sortie : 2026-05-14**

### Faits marquants

- **Renommage du plugin et du namespace.** `id("dev.nucleusframework")`, groupe Maven `dev.nucleusframework`, racine Kotlin `dev.nucleusframework.*`. Voir [migrer depuis 1.x](/docs/migrate/from-1.x).
- **Point d'entrée parapluie `nucleusApplication { }`.** Lock single-instance, livraison des deep-links, init GraalVM, entraînement AOT, priming AutoLaunch et AUMID — tout est automatique.
- **Backend Tao (`decorated-window-tao`).** Chemin sans AWT piloté par la crate Rust `tao`. Wayland natif, multi-touch, pression et inclinaison du stylet, popups natifs, drag-and-drop avec diagnostics, `NativeView` embarqué pour SwiftUI / WebView2 / GTK.
- **`DecoratedWindow` / `DecoratedDialog` comme extensions de `NucleusApplicationScope`** — dispatch indépendant du backend, `MaterialDecoratedWindow` et `JewelDecoratedWindow` suivent le même schéma. CompositionLocals pontés à travers la frontière de scène Tao.
- **Liquid Glass macOS via patch `vtool`.** `LC_BUILD_VERSION` fixé au SDK 26 sur le launcher pour que n'importe quel JDK puisse activer Liquid Glass au runtime — plus besoin d'un JDK compilé avec Xcode 26. Répertoires `.icon` en couches pris en charge sur macOS 26+.
- **GraalVM Native Image : alpha.** Analyseur de bytecode statique pour les métadonnées de réflexion, `CleanupGraalvmMetadataTask`, métadonnées L1 par bibliothèque avec filtrage conditionnel au classpath. Pipeline d'empaquetage de bout en bout (DMG, NSIS, DEB) pour les images natives.

## v1.11.0

**Sortie : 2026-04-13**

### Nouveaux modules

- **Notification Common** (`nucleus.notification-common`) — API de notification multiplateforme unifiée pour Linux, Windows et macOS derrière un seul DSL. Prend en charge les callbacks par notification, jusqu'à 5 boutons d'action, les pièces jointes image et la gestion du rejet.

### Nouvelles fonctionnalités

- **Disposition réactive des boutons de titlebar GNOME** — Les fenêtres décorées sous Linux lisent désormais `org.gnome.desktop.wm.preferences` → `button-layout` via GSettings (`libgio` dlopen) pour déterminer quels boutons afficher et de quel côté. La disposition se met à jour de façon réactive quand l'utilisateur la change dans GNOME Tweaks ou via `gsettings set`. Retombe sur la disposition par défaut sous KDE et les autres environnements de bureau. Nouveau composable `rememberLinuxButtonLayout()` pour un accès direct.
- **Métadonnées de reachability GraalVM pour FileKit et dbus-java** — Les applications qui utilisent FileKit sous Linux n'ont plus besoin d'entrées de reachability manuelles pour les dialogues de fichiers xdg-desktop-portal. Inclut de nouvelles métadonnées conditionnelles pour dbus-java et des internes du JDK Linux (`UnixSystem`, `NativePRNG$NonBlocking`, `CollationData`).
- **Politiques de raccourci pour les notifications Windows** — Nouvel enum `ShortcutPolicy` sur `WindowsNotificationCenter` pour un contrôle plus fin de la création du raccourci dans le menu Démarrer.
- **Propriétés `NucleusApp.appName` et `NucleusApp.aumid`** — Exposent le nom de l'application et l'AUMID pour faciliter la configuration.

### Corrections de bugs

- **Correction de la couleur d'accentuation multicolore macOS** — `system-color` renvoie désormais `null` quand macOS est en mode multicolore, au lieu de renvoyer par erreur le bleu par défaut.
- **Correction des builds incrémentaux dans les tâches GraalVM** — Désactive le suivi d'état sur les modifications du répertoire de sortie partagé pour éviter les builds obsolètes.
- **Correction de la sûreté au null dans les tâches d'application JVM** — Simplifie la gestion du classpath runtime et des arguments `JavaExec` en supprimant le chaînage optionnel superflu.

### Améliorations

- **Chargement plus rapide des bibliothèques natives** — `NativeLibraryLoader` utilise désormais des empreintes CRC-32 pour valider le cache, éliminant les I/O inutiles lors des vérifications.

### Documentation

- **Statut GraalVM passé en alpha** — La prise en charge de GraalVM Native Image est désormais étiquetée « alpha » au lieu d'« expérimentale », reflétant les métadonnées de reachability centralisées désormais livrées par défaut.
- **Nouvelle documentation System Tray** — Section de documentation complète pour ComposeNativeTray avec captures d'écran et GIFs de démo.
- **Réécriture de la landing page** — Repositionne Nucleus comme un successeur natif d'Electron, avec des comparaisons de performances.

## v1.10.0

**Sortie : 2026-04-12**

### Nouveaux modules

- **System Info** (`nucleus.system-info`) — Module d'informations système multiplateforme avec des implémentations natives JNI pour Linux, Windows et macOS. Expose en temps réel les métriques CPU, mémoire et GPU.
    - **Détection GPU et métriques en direct** — Température, utilisation, VRAM, fréquences, consommation et vitesse des ventilateurs sur toutes les plateformes
    - **macOS** : IOKit et SMC pour les métriques GPU, prise en charge d'Apple Silicon et des GPU dédiés
    - **Windows** : DXGI pour l'énumération des GPU, NVIDIA NVML + AMD ADL2 + Intel IGCL pour les métriques en direct, capteurs de zone thermique WMI, données de performance pour la fréquence CPU en temps réel
    - **Linux** : prise en charge des GPU NVIDIA NVML, AMD et Intel
    - Inclut une application de démo (`system-info-demo`) avec des graphiques lets-plot pour l'historique de température CPU

### Corrections de bugs

- **Correction de `latest.yml` non généré pour les formats MSI et Portable** — La génération du YML était limitée à NSIS ; elle couvre désormais aussi les installeurs MSI et Portable
- **Correction de la précision de `releaseDate` dans `latest.yml`** — Utilise une précision à la milliseconde pour correspondre au format electron-builder attendu par l'auto-updater

### CI

- **Ajout des builds natifs `system-info` à la CI** — Compile, vérifie et téléverse les artefacts natifs pour toutes les plateformes dans les workflows `build-natives`, `pre-merge`, `publish-maven`, `publish-plugin`, `test-packaging` et `test-graalvm`
- **Ajout d'une étape de configuration MSVC pour les builds natifs Windows** — Garantit la disponibilité du compilateur Visual Studio en CI pour la compilation JNI Windows

## v1.8.0

- **Notification macOS** (`nucleus.notification-macos`) — Mapping complet de l'API UserNotifications via JNI. Prend en charge les notifications riches avec titre, sous-titre, corps, son, badge, catégories avec actions et planification de la livraison. - **Menu macOS** (`nucleus.menu-macos`) — API native de barre de menu macOS via JNI. Inclut les métadonnées de reachability GraalVM pour les callbacks JNI.

- **SF Symbols** (`nucleus.sf-symbols`) — Module d'intégration des SF Symbols d'Apple.

## v1.7.0

**Sortie : 2026-03-24**

### Nouvelles fonctionnalités

- **Analyseur de bytecode statique pour les métadonnées de réflexion GraalVM** — Nouvelle `AnalyzeStaticMetadataTask` qui scanne le bytecode compilé pour détecter automatiquement les classes nécessitant une configuration de réflexion, JNI et ressources. Élimine l'essentiel de l'écriture manuelle des métadonnées :
    - Détection des callbacks JNI — trouve les méthodes natives et leurs types de paramètres/retour
    - Analyse des ressources — détecte les appels `getResource`/`getResourceAsStream`
    - Wrappers de chargement de classe — identifie les patterns `Class.forName`, `MethodHandles.Lookup.findClass`
    - Résolution des superclasses JNI — inclut les classes parentes nécessaires à l'accès aux champs
    - Entrées de classe enrichies — ajoute les méthodes et champs non natifs aux métadonnées de classe JNI

- **`CleanupGraalvmMetadataTask`** — Nouvelle tâche Gradle qui retire de ton `reachability-metadata.json` manuel les entrées déjà couvertes par les métadonnées de bibliothèque Nucleus (L1/L2/L3). Garde la config manuelle minimale.

- **Prise en charge des associations de fichiers pour l'image native GraalVM sous macOS** — Enregistre les associations de type de fichier dans le bundle `.app` de l'image native pour que macOS ouvre les fichiers avec ton application.

- **Patch de la target de déploiement et de la version du SDK macOS** — Patche automatiquement la target de déploiement Mach-O et la version du SDK dans les binaires d'image native GraalVM pour la compatibilité Liquid Glass macOS.

- **Métadonnées L1 par bibliothèque** — Découpe le fichier de métadonnées L1 monolithique en fichiers par bibliothèque avec filtrage conditionnel, pour n'inclure que les métadonnées pertinentes selon les dépendances réelles du classpath.

- **Métadonnées GraalVM JNA** — Ajoute les exigences JNI de base de JNA aux métadonnées L1.

- **Métadonnées de drag-and-drop AWT et de handler d'ouverture de fichier** — Ajoute les métadonnées de reachability GraalVM pour les classes de drag-and-drop AWT et les handlers d'ouverture de fichier macOS.

- **Métadonnées de ressources Ktor CIO, SLF4J et JDK courantes** — Entrées L1 supplémentaires pour le moteur Ktor CIO, SLF4J et les classes internes du JDK couramment utilisées.

### Corrections de bugs

- **Correction de la duplication des métadonnées de l'agent** — Déduplique la sortie de l'agent de tracing par rapport au Oracle Reachability Metadata Repository et aux résultats de l'analyse statique
- **Correction de la sérialisation du configuration cache dans `runWithNativeAgent`** — Résout les erreurs de sérialisation Gradle quand le configuration cache est activé
- **Correction du pattern de ressources Aqua LAF** — Ajoute un pattern glob pour les ressources du Look-and-Feel Aqua afin d'éviter la duplication par l'agent sous macOS
- **Correction de la version de fichier de classe non prise en charge dans `NativeMethodDetector`** — Gère en douceur les versions de fichier de classe plus récentes au lieu de crasher
- **Correction des faux positifs de `JarResourceDetector`** — Restreint la correspondance de chemins pour éviter de détecter des ressources sans rapport

### Changements incompatibles

- **Répertoire de config GraalVM par défaut modifié** — Le répertoire par défaut des métadonnées GraalVM manuelles est désormais `graalvm/` au lieu de `resources/`. Les projets existants doivent déplacer leur `reachability-metadata.json` vers le nouvel emplacement.

## v1.6.2

**Sortie : 2026-03-22**

### Nouvelles fonctionnalités

- **Propriété `appName`** — Nouvelle propriété de premier niveau dans `nativeDistributions {}` pour le nom d'affichage lisible de l'application (titre de l'installeur, `Name` du `.desktop`, entrée du menu Démarrer). Sépare le nom d'affichage du `packageName`, qui reste l'identifiant technique utilisé pour le nommage de l'exécutable et des fichiers de package.

```kotlin
nativeDistributions {
    appName = "My App"            // Nom d'affichage (installeur, .desktop, menu Démarrer)
    linux { packageName = "myapp" }     // Technique : exécutable, nom du fichier .deb
    windows { packageName = "MyApp" }   // Technique : nom du .exe, MSI
}
```

### Corrections de bugs

- **Correction de l'échec du build MSI avec WiX `LGHT0094: File:mainExecutable not found`** — Le chemin d'empaquetage JVM définit désormais `executableName` sur la tâche electron-builder, comme le faisait déjà le chemin GraalVM. Sans cela, electron-builder retombait sur un nom en minuscules incohérent issu de `package.json`, ce qui faisait échouer WiX.
- **Correction du fichier .deb Linux et du launcher utilisant le nom de projet Gradle au lieu de `linux.packageName`** — `executableName` se résout désormais vers le `packageName` spécifique à la plateforme (`linux.packageName`, `windows.packageName`, `macOS.packageName`) au lieu du nom de projet Gradle.
- **Correction de `productName` absent dans le YAML electron-builder** — Quand aucun `packageName` de premier niveau n'était défini, `productName` était omis de la config générée. Il retombe désormais sur `appName`, puis `packageName`, puis `executableName`.
- **Correction du champ `name` de `package.json` ignorant le nom de package spécifique à la plateforme** — Le `package.json` généré utilise désormais `executableName` (spécifique à la plateforme) pour le champ `name`, afin que la variable `${name}` du template de nom d'artefact se résolve correctement.

## v1.6.0

**Sortie : 2026-03-22**

### Nouveaux modules

- **Taskbar Progress** (`nucleus.taskbar-progress`) — Barre de progression de barre des tâches/dock native et demandes d'attention sur toutes les plateformes. Affiche la progression d'un téléchargement, l'état d'un build ou toute opération longue directement dans la barre des tâches de l'OS. Voir [Taskbar Progress](/docs/lifecycle/taskbar-progress).
    - **Windows** : `ITaskbarList3` (valeur/état de progression) + `FlashWindowEx` (attention)
    - **macOS** : `NSDockTile` avec overlay `NSProgressIndicator` personnalisé + `NSApplication.requestUserAttention`
    - **Linux** : D-Bus `com.canonical.Unity.LauncherEntry` (GNOME, KDE Plasma et environnements compatibles)
    - Cinq états : `NO_PROGRESS`, `INDETERMINATE`, `NORMAL`, `ERROR`, `PAUSED`
    - Demandes d'attention : `INFORMATIONAL` (bref flash) et `CRITICAL` (jusqu'au focus)

- **Métadonnées d'application au runtime** (`NucleusApp`) — Nouveau singleton dans `core-runtime` qui expose au runtime les métadonnées injectées par le plugin : `appId`, `version`, `vendor`, `description`. Alimenté via des propriétés système et une ressource classpath `nucleus-app.properties` générée. Voir [API runtime](/docs/lifecycle/app-metadata).

### Nouvelles fonctionnalités

- **Métadonnées d'image native GraalVM centralisées** — Nucleus livre désormais toutes les métadonnées de réflexion génériques et spécifiques à la plateforme prêtes à l'emploi, organisées en trois niveaux :
    - **L1 (JAR graalvm-runtime)** — Entrées de réflexion génériques multiplateforme pour Compose Desktop, AWT/Swing, Skiko, les fournisseurs de sécurité, les gestionnaires de polices, et plus (~300+ types). Récupérées automatiquement depuis le classpath par native-image.
    - **L2 (Oracle Reachability Metadata Repository)** — Résolution automatique des métadonnées pour toutes les dépendances du classpath runtime depuis le [Oracle GraalVM Reachability Metadata Repository](https://github.com/oracle/graalvm-reachability-metadata). Couvre les bibliothèques populaires comme ktor, kotlinx.serialization, SLF4J, Logback et bien d'autres. Activé par défaut.
    - **L3 (métadonnées de plateforme du plugin)** — Métadonnées AWT/Java2D/police/sécurité spécifiques à la plateforme pour macOS, Windows et Linux, livrées à l'intérieur du plugin Gradle. Écrites dans le répertoire de build à la compilation — aucun bloc `when` par plateforme nécessaire dans ton script de build.

    Tu n'as plus besoin de copier des milliers d'entrées de réflexion depuis l'application d'exemple. La plupart des applications fonctionneront sans aucune configuration de réflexion manuelle. Voir [Métadonnées de réflexion centralisées](/docs/performance/graalvm#centralized-reflection-metadata).

- **DSL `metadataRepository {}`** — Nouveau bloc de configuration dans `graalvm {}` pour contrôler l'intégration du Oracle Reachability Metadata Repository. Prend en charge `enabled`, `version`, `excludedModules` et `moduleToConfigVersion`. Activé par défaut avec la version `0.10.6`.

- **Tâche `resolveReachabilityMetadata`** — Nouvelle tâche Gradle qui résout les entrées du Oracle Reachability Metadata Repository pour toutes les dépendances du classpath runtime. S'exécute automatiquement avant `packageGraalvmNative`.

- **Inclusion automatique des ressources Compose Multiplatform** — `graalvm-runtime` inclut désormais `composeResources/.*` dans les patterns de ressources de `native-image.properties`, donc toutes les ressources Compose (images, chaînes, polices chargées via `Res.*`) sont incluses automatiquement.

- **Fusion et déduplication de l'agent** — La tâche `runWithNativeAgent` déduplique désormais la sortie de l'agent par rapport aux métadonnées de bibliothèque déjà livrées dans les JARs du classpath. Cela empêche l'agent de tracing de réajouter des entrées que Nucleus (ou d'autres bibliothèques) fournissent déjà, gardant la config spécifique à ton application propre et minimale.

- **Enum `UpdateLevel`** — `UpdateResult.Available` porte désormais un `UpdateLevel` (`MAJOR`, `MINOR`, `PATCH`, `PRE_RELEASE`) calculé en comparant les numéros de version sémantique. Permet à l'UI d'adapter son message selon l'importance de la mise à jour. Voir [Auto Update](/docs/packaging/auto-update#update-level).

- **Détection post-mise à jour** — Les nouvelles méthodes `NucleusUpdater.consumeUpdateEvent()` et `wasJustUpdated()` détectent que l'application vient d'être mise à jour et renvoient un `UpdateEvent` avec `previousVersion`, `newVersion` et `updateLevel`. - **Modules jlink par défaut étendus** — `java.net.http` et `jdk.accessibility` sont désormais inclus dans la liste des modules jlink par défaut aux côtés de `java.base`, `java.desktop`, `java.logging` et `jdk.crypto.ec`. Plus besoin de les ajouter manuellement.

## v1.3.0

**Sortie : 2026-03-02**

### Nouvelles fonctionnalités

- **Prise en charge de GraalVM Native Image (expérimental)** — Compile les applications Compose Desktop en binaires natifs autonomes avec un cold boot rapide (~0,5 s), une empreinte mémoire moindre (~100–150 Mo contre ~300–400 Mo sur JVM) et des bundles plus petits. Nouveau bloc DSL `graalvm {}`, tâche `runWithNativeAgent` pour la collecte des métadonnées de réflexion, et tâches d'empaquetage `packageGraalvmNative` / `packageGraalvmDeb` / `packageGraalvmDmg` / `packageGraalvmNsis`. Nécessite [BellSoft Liberica NIK 25](https://bell-sw.com/liberica-native-image-kit/). Voir [GraalVM Native Image](/docs/performance/graalvm).
- **Nouveau module `graalvm-runtime`** (`nucleus.graalvm-runtime`) — Centralise la logique de bootstrap de l'image native dans un seul appel `GraalVmInitializer.initialize()` : Metal L&F, configuration de `java.home`/`java.library.path`, init précoce du charset/fontmanager, détection HiDPI Linux et substitutions de police GraalVM `@TargetClass` pour Windows et Linux.
- **Découpage du module Decorated Window** — `decorated-window` découpé en `decorated-window-core`, `decorated-window-jbr` (basé sur JBR, même comportement qu'avant) et `decorated-window-jni` (sans JBR, fonctionne avec GraalVM). Voir le [Guide de migration](#migration-guide-12x--13x) ci-dessous.
- **Module `decorated-window-jni`** — Nouvelle implémentation JNI de `DecoratedWindow` qui fonctionne sans JetBrains Runtime, y compris la prise en charge de Linux via un pont JNI natif. Compatible avec GraalVM Native Image.
- **Prise en charge du scaling HiDPI Linux** — Gestion native de `GDK_SCALE` pour un rendu correct sur les affichages Linux HiDPI.
- **Notarisation automatique des distributions macOS** — `packageDistributionForCurrentOS` notarise désormais automatiquement sous macOS quand les identifiants de notarisation sont configurés.
- **Application Jewel Sample** — Vitrine d'UI Jewel autonome avec CI d'image native GraalVM, empaquetage spécifique à la plateforme et signature de code Windows.

### Corrections de bugs

## Guide de migration : 1.2.x → 1.3.x

### Decorated window : découpage du module monolithique

Le module `decorated-window` a été découpé en trois modules :

| Avant (1.2.x) | Après (1.3.x) |
|---|---|
| `nucleus.decorated-window` | `nucleus.decorated-window-core` (partagé) |
| | `nucleus.decorated-window-jbr` (implémentation JBR) |
| | `nucleus.decorated-window-jni` (implémentation JNI, nouveau) |

**Mise à jour de la dépendance** — remplace :

```kotlin
implementation("dev.nucleusframework:nucleus.decorated-window:<version>")
```

Par l'une de :

```kotlin
// Basé sur JBR (même comportement qu'avant)
implementation("dev.nucleusframework:nucleus.decorated-window-jbr:<version>")

// Basé sur JNI (sans dépendance JBR, fonctionne avec GraalVM)
implementation("dev.nucleusframework:nucleus.decorated-window-jni:<version>")
```

**Changements incompatibles dans `TitleBarColors`** — les champs suivants ont été **supprimés** :

- `titlePaneButtonHoveredBackground`
- `titlePaneButtonPressedBackground`
- `titlePaneCloseButtonHoveredBackground`
- `titlePaneCloseButtonPressedBackground`

Ces couleurs d'état de bouton spécifiques à la plateforme sont désormais gérées en interne par l'implémentation native de chaque module. Si tu construisais `TitleBarColors` explicitement avec ces champs, retire-les.

**Aucun autre changement de code requis** — toutes les API composables (`DecoratedWindow`, `DecoratedDialog`, `TitleBar`, `DialogTitleBar`), scopes et types d'état sont identiques. Aucun changement d'import nécessaire — le package reste `dev.nucleusframework.window`.

Voir [Decorated Window](/docs/window) pour tous les détails sur le choix entre JBR et JNI.

## v1.0.0

**Sortie : 2026-02-17**

Première release — fork du plugin Gradle JetBrains Compose Desktop, réempaqueté sous le nom **Nucleus** avec des fonctionnalités d'empaquetage, de distribution et de runtime étendues.

### Empaquetage et distribution

- **Backend electron-builder** — Formats DMG, PKG, NSIS, NSIS Web, AppImage, DEB, RPM, Snap et AppX. Remplace jpackage pour la génération d'installeurs.
- **Signature de code et notarisation macOS** — Signature Developer ID et App Store, notarisation automatique via `notarytool`.
- **App Sandbox macOS** — Builds PKG sandboxed avec extraction de bibliothèque JNI, gestion des entitlements et profils de provisioning.
- **Signature de code Windows** — Intégration SignTool avec hachage SHA-256.
- **Empaquetage AppX** — Empaquetage Windows Store avec assets de tuile et configuration d'identité.
- **Personnalisation DMG** — Images de fond, positionnement des icônes, taille de fenêtre, icônes de badge et sélection de format.
- **Icônes en couches** — Prise en charge des répertoires `.icon` macOS 26+ pour les effets dynamiques de tilt/profondeur.
- **Écran de démarrage** — Prise en charge d'une image de splash dans le launcher JVM.
- **Binaires macOS universels** — Workflow CI pour les binaires fat basés sur `lipo` (Apple Silicon + Intel).
- **DSL typé** — Des enums pour les formats de target, les modes de publication et les réglages de compression remplacent les chaînes brutes.
- **Options d'empaquetage Linux** — `startupWMClass`, `debDepends`, `rpmRequires`, `debCompression`, `rpmCompression`, scripts après installation et suffixes d'architecture.
- **Associations de fichiers** — Injection `CFBundleURLTypes` macOS et prise en charge des associations de fichiers Linux/Windows via electron-builder.

### Modules runtime

- **`core-runtime`** — Détection du type d'exécutable, identification de plateforme, `SingleInstanceManager`, `DeepLinkHandler`.
- **`aot-runtime`** — Détection du mode de cache AOT pour JDK 25+ (Project Leyden). Tâche de génération de cache AOT avec timeout de sécurité.
- **`updater-runtime`** — Moteur d'auto-update avec providers GitHub et S3. Vérification SHA-512, suivi de progression, `installAndRestart()`. Stratégies d'installation spécifiques à la plateforme : script shell détaché sous macOS, PowerShell en arrière-plan sous Windows, remplacement AppImage sous Linux.

### Build et CI

- **Actions composites** — Actions `setup-nucleus`, `update-yml` et de publication de release.
- **Workflow de release multiplateforme** — macOS (arm64 + x64 + universel), Windows (x64 + arm64), Linux (x64 + arm64).
- **Prise en charge du configuration cache** — Compatibilité complète avec le configuration cache Gradle.
- **Version du plugin depuis le tag Git** — Aucune gestion manuelle de version dans `gradle.properties`.
