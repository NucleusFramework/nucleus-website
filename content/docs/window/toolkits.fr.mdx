Nucleus livre des implémentations Compose first-party de chaque grand langage de design desktop. Le même `DecoratedWindow`, restylé par le module de toolkit de le choix — l'utilisateur Mac voit macOS, l'utilisateur Windows voit Fluent, l'utilisateur Ubuntu voit Yaru, et un outil type IDE prend Jewel. Plus de Material peinturé sur Win32.

## TL;DR
- Le même `DecoratedWindow` partout — les toolkits ne changent que le style visuel.
- Un par OS, ou mélange libre : chaque toolkit fonctionne sur chaque plateforme.
- Tu changes de toolkit en changeant la dépendance Gradle, pas le UI.
- Material 2 et Material 3 restent disponibles comme repli.

| Capacité | macOS 26 | Fluent | Yaru | Jewel | Material 3 |

| Look natif sur l'OS hôte | macOS | Windows | Ubuntu | N'importe (IDE) | N'importe |

## Comment ça marche

Chaque module de toolkit enveloppe le même `DecoratedWindow` de `nucleus-application` et substitue ses propres `TitleBarStyle` et `DecoratedWindowStyle`. Les Composables de les écrans ne changent pas — seul le wrapper en haut de le arbre change. C'est pour ça que mixer les toolkits est bon marché : on livres un build Fluent sur macOS pour les screenshots, on livres Liquid Glass aux utilisateurs, avec le même code.

Les styles vivent dans `decorated-window-core` en données plates (`TitleBarColors`, `TitleBarMetrics`, `DecoratedWindowMetrics`). Chaque module de toolkit expose un Composable `remember*Style()` qui les dérive de son thème source (`JewelTheme` pour Jewel, `MaterialTheme` pour Material, etc.).

<Callout type="info">
Les toolkits stylisent le chrome de la fenêtre. Les widgets à l'intérieur (boutons, listes, champs texte) restent de le choix — Material 3, Jewel, ou le propre design system. Mixe librement.
</Callout>
