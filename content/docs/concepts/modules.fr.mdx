Nucleus est un ensemble de modules Gradle versionnés séparément. Un module ombrelle apporte
les essentiels du runtime, et chaque autre capacité est un module distinct que tu ajoutes au
besoin, si bien que ton build n'embarque que ce que ton application utilise.

## Le module ombrelle

Ajoute `nucleus-application` pour tirer les essentiels du runtime :

```kotlin title="build.gradle.kts"
dependencies {
    implementation("dev.nucleusframework:nucleus.nucleus-application:2.0.0")
}
```

Cela tire trois modules runtime :

- `core-runtime` — détection de plateforme, chargement de bibliothèques natives, gestionnaire de deep links, verrou single-instance et métadonnées d'application.
- `aot-runtime` — détection du mode cache AOT.
- `graalvm-runtime` — les substitutions SVM et l'initialiseur appelé depuis `nucleusApplication { }`.

Il apporte aussi le code Compose de fenêtre partagé dans `decorated-window-core` ; tu n'ajoutes
donc qu'un backend de fenêtre par-dessus. Tout le reste est optionnel.

## Modules de fenêtre

Chaque application utilise exactement un backend de fenêtre, plus un adaptateur de style
optionnel.

| Module | Rôle |
|---|---|
| `decorated-window-core` | Styling, thèmes et jeux d'icônes Compose partagés. Tiré par les autres. |
| `decorated-window-tao` | Backend natif Rust ([Tao](/docs/concepts/backends)). Par défaut pour les nouveaux projets. |
| `decorated-window-jbr` / `decorated-window-jni` / `decorated-window-awt` | Backend AWT. **Déprécié**, sera retiré dans une prochaine version de Nucleus. |
| `decorated-window-jewel` | Adaptateur de style — look IntelliJ Platform. |
| `decorated-window-material2` | Adaptateur de style — Material 2. |
| `decorated-window-material3` | Adaptateur de style — Material 3. |

Choisis exactement un backend (`-tao`, ou l'un des modules AWT dépréciés), et zéro ou un
adaptateur de style.

## Ajouter un module

Déclare l'ombrelle, un backend de fenêtre et les modules de capacité dont tu as besoin :

```kotlin title="build.gradle.kts"
val nucleusVersion = "2.0.0"

dependencies {
    implementation("dev.nucleusframework:nucleus.nucleus-application:$nucleusVersion")
    implementation("dev.nucleusframework:nucleus.decorated-window-tao:$nucleusVersion")

    // Ajoute les capacités dont tu as besoin
    implementation("dev.nucleusframework:nucleus.notification-common:$nucleusVersion")
    implementation("dev.nucleusframework:nucleus.global-hotkey:$nucleusVersion")
    implementation("dev.nucleusframework:nucleus.system-info:$nucleusVersion")
}
```

<Callout type="info">
Épingle tous les artefacts `nucleus.*` à la même version. Les modules sont publiés ensemble.
</Callout>

## Fonctionnement de la composition des modules

La plupart des modules dépendent de `core-runtime` pour la détection de plateforme et les
métadonnées d'application. Quelques catalogues autonomes, comme `sf-symbols` et
`freedesktop-icons`, n'ont aucune dépendance runtime. Certains modules dépendent les uns des
autres : `notification-common` orchestre les modules de notification par OS, et
`taskbar-progress-tao` dépend de `nucleus-application` pour ajouter des fonctions d'extension
sur `NucleusWindow`.

Le plugin Gradle injecte `nucleus-app.properties` dans le classpath au moment du build, si
bien que n'importe quel module lit l'identité de ton application via `NucleusApp.appId`,
`NucleusApp.version`, `NucleusApp.aumid` et les autres propriétés, sans câblage
supplémentaire.

Lors du build GraalVM, les métadonnées de reachability de chaque module sont intégrées
automatiquement : aucune configuration de réflexion à maintenir à la main.
