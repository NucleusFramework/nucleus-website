Kotlin tourne déjà sur Android, sur iOS via Kotlin/Native, et sur le web via Kotlin/JS et Wasm. Sur le desktop, tu peux afficher une fenêtre Compose, mais atteindre le système d'exploitation supposait jusqu'ici de compiler des bibliothèques natives par plateforme, de câbler des ponts JNI ou FFM, et d'apprendre une API native différente pour chaque OS. Nucleus te permet de construire toute l'application desktop en Kotlin — UI, logique métier, appels OS et packaging — dans un seul process.

Cette page compare cette approche à Electron, Tauri et Compose Multiplatform vanilla (CMP).

## Comparaison

| | **Nucleus** | Electron | Tauri | CMP par défaut |
|--|------------|----------|-------|----------------|
| Langage(s) | Kotlin | JS + Node (+ C++ pour le natif) | JS + Rust | Kotlin |
| UI | Compose + Skia GPU | Chromium (Blink + V8) | WebView OS | Compose + Skia GPU |
| API OS | 30+ modules Kotlin, mono-process | Node `child_process` / modules natifs | Commandes Rust via IPC | Limité (`Tray`, `Notification`, pas de fenêtre décorée) |
| Formats de packaging | **18** (DMG, PKG, Exe, MSI, NSIS, NSIS Web, Portable, AppX, DEB, RPM, AppImage, RawAppImage, Pacman, Snap, Flatpak, ZIP, TAR, 7Z) | 5–7 via electron-builder | 5 via tauri-bundler | 6 (DMG, PKG, MSI, EXE, DEB, RPM) |
| Auto-update | Intégré (`updater-runtime`) | electron-updater | Plugin Tauri Updater | Aucun |
| Cold start | ~0,5 s (GraalVM) · ~1,0 s (JVM+AOT) | 2–3 s | &lt; 1 s | 1–2 s |
| RAM au repos | 60–150 MB | 200–500 MB | 50–80 MB | 200 MB |
| Taille binaire | 38 MB (GraalVM) · ~120 MB (JVM) | 150–200 MB | 5–20 MB | 80–120 MB |
| Décorations natives | Oui — Liquid Glass, Fluent, Yaru, Jewel | Limité (chrome Chromium) | Oui (chrome OS) | Limité |
| Modèle | Un langage bout en bout | Deux runtimes, pont IPC | Frontend web + backend Rust | Un langage, moins d'éléments intégrés |

## Un seul langage sur toute la pile

Electron et Tauri répartissent la pile sur plusieurs langages. Tu écris l'UI en JavaScript, les parties natives en Rust, C++ ou Swift, et tu les relies par un pont IPC. Nucleus garde toute la pile en Kotlin — UI, logique métier, appels OS et packaging — avec le même call graph, le même modèle mémoire et le même debugger.

Quand tu as besoin de descendre au niveau d'une plateforme — intégrer une vue SwiftUI, appeler Win32 directement, ou utiliser une interface D-Bus propre à Linux — le [backend Tao](/docs/tao) expose `NativeView`, et [native-access](/docs/performance/native-access) fournit du FFI typé sans quitter le source Kotlin.

## Quand choisir chaque outil

- **Electron** — l'écosystème et le bassin d'embauche les plus larges. Choisis-le quand la vitesse de livraison et la familiarité avec JavaScript comptent plus que la RAM, la taille du binaire ou le rendu natif.
- **Tauri** — les plus petits binaires de cette comparaison. Choisis-le quand tu acceptes la WebView propre à chaque plateforme et le changement de contexte Rust/JavaScript.
- **CMP vanilla** — la fondation sur laquelle repose Nucleus. Choisis-le quand tu veux Compose sans la surface supplémentaire qu'ajoute Nucleus, et que tu es prêt à écrire toi-même le packaging, la signature, l'updater et la fenêtre décorée.
- **Nucleus** — choisis-le quand tu veux un seul langage sur toute la pile, des décorations de fenêtre natives, l'écosystème JVM in-process, et une pipeline de packaging qui couvre 18 formats et quatre stores.

## Notes

Les chiffres de performance ci-dessus sont typiques d'une application de type Hello-World. Le cold start et la RAM réels varient selon ce que ton application charge. Nucleus dispose de deux cibles runtime : GraalVM native image pour le cold start et la taille du binaire, et JVM + cache AOT pour le débit. Voir [performance](/docs/performance) pour les détails.

<Callout type="info">
Les chiffres d'Electron et de Tauri proviennent de benchmarks publics, et non de mesures effectuées dans ce dépôt.
</Callout>
