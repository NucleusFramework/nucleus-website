Electron renders desktop UI with a bundled Chromium and runs application logic in a
separate Node process, communicating over IPC. Nucleus renders Compose UI to Skia and runs
the UI, business logic, and OS access in a single Kotlin process. This page compares the two.

## Comparison

| | **Nucleus** | Electron |
|--|------------|----------|
| Languages | Kotlin | JS + Node + native modules |
| Renderer | Compose UI + Skia GPU | Chromium (Blink + V8) |
| Process model | Single process | Renderer + main + IPC |
| Native APIs | 30+ Kotlin modules, in-process | Node `child_process` / native modules |
| OS chrome | Decorated windows: Liquid Glass, Fluent, Yaru, Jewel | Chromium-style chrome |
| Cold start | ~0.5 s (GraalVM) · ~1.0 s (JVM+AOT) | 2–3 s |
| RAM idle | 60–150 MB | 200–500 MB |
| Binary size | 38 MB (GraalVM) · ~120 MB (JVM) | 150–200 MB |
| Auto-update | Built-in (`updater-runtime`) | electron-updater |
| Packaging | 16 formats from one DSL | 5–7 via electron-builder |
| Ecosystem | JVM (Maven Central, ~500k libs) | npm |

<Callout type="info">
Memory and cold-start figures vary widely by app. The Electron ranges above are a typical
"Hello World plus a small single-page app" envelope from public benchmarks, not fixed values.
</Callout>

## Rendering

Electron bundles a full browser engine in every app: V8, Blink, and Skia, plus a Node
runtime, all linked into the process tree. Each running app carries its own copy, so memory
use adds up across several apps.

Nucleus renders Compose UI to Skia through the Compose Multiplatform Skiko backend. There is
no Blink, V8, or Node. The JVM runs your code and Skia draws to the GPU, all in a single
process sharing one heap. After HotSpot warms up, throughput on hot paths is comparable to
native code.

## OS access

Electron's renderer process cannot call the OS directly. It sends messages to the main
process over `ipcRenderer` and `ipcMain`, so every file read, tray click, and notification
crosses a serialization boundary. Type safety stops at that boundary, and each feature is
split across two processes with `contextBridge` wrappers and two sets of devtools.

In Nucleus, your Compose UI calls `NotificationManager.send(...)`,
`GlobalHotKeyManager.register(...)`, or `SystemInfo.cpuInfo()` directly — same call graph,
same coroutine, same debugger.

## Window decorations

Electron windows render with Chromium's chrome. Matching a native look means replacing the
frame, applying vibrancy, and encoding draggable regions in CSS. The Nucleus
[decorated window](/docs/window) layer draws the title bar inside Compose with four toolkits:

- macOS — Liquid Glass on SDK 26+, applied automatically through `vtool` patching.
- Fluent — Windows 11 acrylic, snap-layout hints, and mica.
- Yaru — Ubuntu button layout that follows GNOME settings live.
- Jewel — IntelliJ Platform look.

Material 2 and Material 3 adapters ship for use on top.

## Where Electron fits better

- **Hiring pool.** More developers know JavaScript than Kotlin Multiplatform desktop.
- **Ecosystem age.** Electron ships VS Code, Slack, Discord, Figma, and Notion, with a
  decade of production use.
- **Web reuse.** If your product is already a single-page app, wrapping it in Electron is
  less work than rewriting it in Compose for a first release.
- **Node libraries.** npm covers a larger surface in some categories, though the JVM has more
  artifacts overall (~500k on Maven Central).

## Where Nucleus fits better

- One language across UI, business logic, OS calls, and packaging — one call graph, one debugger.
- Native rendering rather than a bundled browser.
- Two runtimes (GraalVM Native Image and JVM + AOT cache) from the same source — see
  [performance](/docs/performance/index).
- 16 packaging formats against 5–7 — see [packaging comparison](/docs/compare/packaging).
- The JVM library catalog runs in-process. Apache Lucene, DuckDB, ONNX Runtime, Tika, Netty,
  and Spark are an `implementation` dependency away, with no native bridge.

## When to choose Electron

You have a JavaScript team shipping a first release, your product already exists as a web
app, and your memory and performance budget can absorb the bundled Chromium. Or you have an
existing Electron codebase where migration costs more than the ongoing overhead.

## When to choose Nucleus

You want one language end to end, native rendering, native installers, and the JVM ecosystem
in-process. You ship to enterprise networks where corporate root CAs matter and `native-ssl`
handles them. Cold start matters enough to wire up GraalVM Native Image.
