Electron rend l'UI desktop avec un Chromium embarqué et exécute la logique applicative dans
un process Node séparé, en communiquant par IPC. Nucleus rend l'UI Compose vers Skia et
exécute l'UI, la logique métier et l'accès à l'OS dans un seul process Kotlin. Cette page
compare les deux.

## Comparaison

| | **Nucleus** | Electron |
|--|------------|----------|
| Langages | Kotlin | JS + Node + modules natifs |
| Rendu | Compose UI + Skia GPU | Chromium (Blink + V8) |
| Modèle de process | Mono-process | Renderer + main + IPC |
| API natives | 30+ modules Kotlin, in-process | Node `child_process` / modules natifs |
| Chrome OS | Fenêtres décorées : Liquid Glass, Fluent, Yaru, Jewel | Chrome Chromium |
| Cold start | ~0,5 s (GraalVM) · ~1,0 s (JVM+AOT) | 2–3 s |
| RAM au repos | 60–150 MB | 200–500 MB |
| Taille binaire | 38 MB (GraalVM) · ~120 MB (JVM) | 150–200 MB |
| Auto-update | Intégré (`updater-runtime`) | electron-updater |
| Packaging | 18 formats depuis un DSL | 5–7 via electron-builder |
| Écosystème | JVM (Maven Central, ~500k libs) | npm |

<Callout type="info">
Les chiffres de mémoire et de cold start varient beaucoup d'une app à l'autre. Les
fourchettes Electron ci-dessus correspondent à une enveloppe typique « Hello World plus
petite single-page app » issue de benchmarks publics, pas à des valeurs fixes.
</Callout>

## Rendu

Electron embarque un moteur de navigateur complet dans chaque app : V8, Blink et Skia, plus
un runtime Node, tous liés dans l'arbre de process. Chaque app en cours porte sa propre
copie, si bien que la consommation mémoire s'accumule sur plusieurs apps.

Nucleus rend l'UI Compose vers Skia via le backend Skiko de Compose Multiplatform. Il n'y a
ni Blink, ni V8, ni Node. La JVM exécute ton code et Skia dessine sur le GPU, le tout dans un
seul process partageant un seul heap. Une fois HotSpot chauffé, le débit sur les hot paths
est comparable à celui du code natif.

## Accès à l'OS

Le process renderer d'Electron ne peut pas appeler l'OS directement. Il envoie des messages
au process main via `ipcRenderer` et `ipcMain`, donc chaque lecture de fichier, chaque clic
dans la system tray et chaque notification traverse une frontière de sérialisation. La
sécurité de type s'arrête à cette frontière, et chaque fonctionnalité est répartie sur deux
process avec des wrappers `contextBridge` et deux jeux de devtools.

Avec Nucleus, ton UI Compose appelle `NotificationManager.send(...)`,
`GlobalHotKeyManager.register(...)` ou `SystemInfo.cpuInfo()` directement — même call graph,
même coroutine, même debugger.

## Décorations de fenêtre

Les fenêtres Electron s'affichent avec le chrome de Chromium. Obtenir un rendu natif oblige à
remplacer le cadre, appliquer la vibrancy et encoder les régions déplaçables en CSS. La
couche [fenêtre décorée](/docs/window) de Nucleus dessine la barre de titre dans Compose avec
quatre toolkits :

- macOS — Liquid Glass au SDK 26+, appliqué automatiquement via le patching `vtool`.
- Fluent — acrylique Windows 11, indices de snap-layout et mica.
- Yaru — disposition de boutons Ubuntu qui suit les réglages GNOME en direct.
- Jewel — look IntelliJ Platform.

Des adaptateurs Material 2 et Material 3 sont fournis pour venir par-dessus.

## Là où Electron convient mieux

- **Bassin d'embauche.** Plus de développeurs connaissent JavaScript que Kotlin Multiplatform desktop.
- **Âge de l'écosystème.** Electron fait tourner VS Code, Slack, Discord, Figma et Notion,
  avec une décennie d'usage en production.
- **Réemploi web.** Si ton produit est déjà une single-page app, l'emballer dans Electron
  demande moins de travail que le réécrire en Compose pour une première version.
- **Bibliothèques Node.** npm couvre une surface plus large dans certaines catégories, même
  si la JVM compte plus d'artefacts au total (~500k sur Maven Central).

## Là où Nucleus convient mieux

- Un seul langage pour l'UI, la logique métier, les appels OS et le packaging — un seul call
  graph, un seul debugger.
- Un rendu natif plutôt qu'un navigateur embarqué.
- Deux runtimes (GraalVM Native Image et JVM + cache AOT) depuis la même source — voir
  [performance](/docs/performance/index).
- 18 formats de packaging contre 5–7 — voir [comparaison packaging](/docs/compare/packaging).
- Le catalogue de bibliothèques JVM tourne in-process. Apache Lucene, DuckDB, ONNX Runtime,
  Tika, Netty et Spark sont à une dépendance `implementation` de distance, sans pont natif.

## Quand choisir Electron

Tu as une équipe JavaScript qui livre une première version, ton produit existe déjà en web
app, et ton budget mémoire et performance peut absorber le Chromium embarqué. Ou tu as un
codebase Electron existant dont la migration coûte plus cher que la charge continue.

## Quand choisir Nucleus

Tu veux un seul langage de bout en bout, un rendu natif, des installeurs natifs et
l'écosystème JVM in-process. Tu livres à des réseaux d'entreprise où les CA racines corporate
comptent et où `native-ssl` les prend en charge. Le cold start compte assez pour câbler
GraalVM Native Image.
