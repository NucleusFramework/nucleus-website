---
title: Nucleus vs Tauri
description: Compare Nucleus and Tauri across language, rendering, process model, and packaging.
---

Tauri and Nucleus share a foundation: both use the OS-native WebView or renderer instead of bundling Chromium, and both build on the Rust [`tao`](https://github.com/tauri-apps/tao) windowing crate, so both ship small binaries with fast cold start. They diverge on language and rendering. Tauri pairs a JavaScript frontend with a Rust backend over IPC. Nucleus runs a single Kotlin process from the UI down to the native windowing layer.

## How they compare

| | **Nucleus** | Tauri |
|--|------------|-------|
| Languages | Kotlin | JS + Rust |
| UI tech | Compose UI + Skia GPU | OS WebView |
| Windowing | Tao (via JNI) | Tao (native Rust) |
| Process model | Single process | Webview process + main, IPC |
| OS APIs | 30+ Kotlin modules in-process | Rust `command`s called from JS |
| Cold start | ~0.5 s (GraalVM) · ~1.0 s (JVM+AOT) | &lt; 1 s |
| RAM idle | 60–150 MB | 50–80 MB |
| Binary size | 38 MB (GraalVM) · ~120 MB (JVM) | 5–20 MB |
| Multi-touch / pen | Yes — `TaoTouchEvent`, pressure, tilt | Limited — WebView pointer events |
| Native chrome | Compose-rendered Liquid Glass / Fluent / Yaru / Jewel | OS chrome |
| Packaging | 16 formats | 5 (DMG, MSI, NSIS, DEB, AppImage) |

## Shared Tao foundation

Tauri's windowing crate `tao` is a fork of `winit` extended for desktop needs — menus, system tray, native popups, drag-and-drop. Nucleus vendors `tao` inside the `decorated-window-tao` module, binds it through JNI, and exposes it to Kotlin:

- `taoApplication { … }` replaces Compose Desktop's AWT-based `application { }`.
- `DecoratedWindow(…)` renders without an AWT frame — native Wayland on Linux, multi-touch, pen pressure and tilt, native popups, and drag-and-drop with diagnostics.
- `NativeView` embeds SwiftUI, WebView2, or GTK widgets inline in Compose.

Tauri uses `tao` to render an OS WebView and route JS calls back through Rust. Nucleus uses `tao` to render Compose UI with Skia, with no WebView in the loop.

## Language model

Tauri splits the stack by language: Rust on the backend, JavaScript on the frontend, and a serialization boundary between them. That means two toolchains and, in practice, often two teams.

Nucleus stays in Kotlin throughout. UI components, the same `ViewModel` you ship on Android, OS calls, and the packaging DSL are one language and one call graph. For teams already working in Kotlin, that removes the frontend/backend split.

## Rendering

Tauri renders through the OS WebView: WebView2 on Windows, WKWebView on macOS, WebKitGTK on Linux. This keeps binaries small and uses hardware acceleration the OS already tunes, at the cost of per-OS rendering differences, CSS for native widgets, and animation smoothness tied to the WebView's compositor.

Nucleus renders Compose UI to Skia via Skiko — Metal on macOS, OpenGL or Direct3D elsewhere — so the renderer is the same on every OS. Pen and trackpad gestures arrive as typed events, and the [Tao backend](/docs/tao) adds native Wayland support on Linux.

## Native window decorations

Tauri's window chrome is whatever the OS WebView renders; custom title bars require CSS that approximates native behavior.

Nucleus ships four [toolkit](/docs/window/toolkits) themes for decorated windows — Liquid Glass (macOS 26+), Fluent (Windows), Yaru (Ubuntu), and Jewel (IntelliJ Platform). Each renders in Compose, is backed by native title-bar geometry, and can be themed from your design system.

## Where Tauri fits

- Binary size: 5–20 MB, smaller than Nucleus at 38 MB (GraalVM) or 120 MB (JVM+AOT).
- JavaScript-first teams that want to keep a web frontend with a Rust backend.
- Access to the Rust crate ecosystem on the backend. Nucleus draws on the JVM ecosystem instead.

## Where Nucleus fits

- One language end-to-end: UI, business logic, OS calls, and packaging in Kotlin.
- Native rendering with Compose and Skia and per-OS window decoration toolkits, with no WebView.
- Multi-touch, pen pressure and tilt, and native Wayland on Linux through the Tao backend.
- The JVM library ecosystem in-process — Lucene, DuckDB, ONNX Runtime, Apache Tika — added as an `implementation` dependency.
- 16 packaging formats against 5 — see the [packaging comparison](/docs/compare/packaging).

## When to choose Tauri

Choose Tauri when you want the smallest possible binary, you're comfortable splitting work across JavaScript and Rust, and your UI is web-shaped so the WebView is a good fit.

## When to choose Nucleus

Choose Nucleus when you want one language end-to-end, native rendering, multi-touch, pen, and Wayland support, and the JVM library ecosystem in-process — especially if you already ship Kotlin to Android, iOS, or the web and want desktop to match.
