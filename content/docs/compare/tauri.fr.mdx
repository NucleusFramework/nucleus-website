---
title: Nucleus vs Tauri
description: Compare Nucleus et Tauri sur le langage, le rendu, le modèle de process et le packaging.
---

Tauri et Nucleus partagent une fondation : tous deux utilisent la WebView ou le renderer natif de l'OS au lieu d'embarquer Chromium, et tous deux reposent sur la crate Rust de windowing [`tao`](https://github.com/tauri-apps/tao), ce qui leur donne à tous deux de petits binaires et un cold start rapide. Ils divergent sur le langage et le rendu. Tauri associe un frontend JavaScript à un backend Rust via IPC. Nucleus fait tourner un seul process Kotlin, de l'UI jusqu'à la couche de windowing native.

## Comparaison

| | **Nucleus** | Tauri |
|--|------------|-------|
| Langages | Kotlin | JS + Rust |
| UI | Compose UI + Skia GPU | WebView OS |
| Windowing | Tao (via JNI) | Tao (Rust natif) |
| Modèle process | Mono-process | Process WebView + main, IPC |
| API OS | 30+ modules Kotlin in-process | `command`s Rust appelées depuis JS |
| Cold start | ~0,5 s (GraalVM) · ~1,0 s (JVM+AOT) | &lt; 1 s |
| RAM idle | 60–150 MB | 50–80 MB |
| Taille binaire | 38 MB (GraalVM) · ~120 MB (JVM) | 5–20 MB |
| Multi-touch / stylet | Oui — `TaoTouchEvent`, pression, tilt | Limité — pointer events WebView |
| Chrome natif | Liquid Glass / Fluent / Yaru / Jewel rendus en Compose | Chrome OS |
| Packaging | 16 formats | 5 (DMG, MSI, NSIS, DEB, AppImage) |

## Fondation Tao partagée

La crate de windowing `tao` de Tauri est un fork de `winit` étendu pour les besoins du desktop — menus, system tray, popups natifs, drag-and-drop. Nucleus vendorise `tao` dans le module `decorated-window-tao`, la branche via JNI et l'expose à Kotlin :

- `taoApplication { … }` remplace l'`application { }` basé sur AWT de Compose Desktop.
- `DecoratedWindow(…)` se rend sans frame AWT — Wayland natif sur Linux, multi-touch, pression et tilt du stylet, popups natifs et drag-and-drop avec diagnostics.
- `NativeView` embarque des widgets SwiftUI, WebView2 ou GTK directement dans Compose.

Tauri utilise `tao` pour rendre une WebView OS et router les appels JS via Rust. Nucleus utilise `tao` pour rendre l'UI Compose avec Skia, sans WebView dans la boucle.

## Modèle de langage

Tauri sépare la stack par langage : Rust côté backend, JavaScript côté frontend, avec une frontière de sérialisation entre les deux. Cela implique deux chaînes d'outils et, en pratique, souvent deux équipes.

Nucleus reste en Kotlin de bout en bout. Les composants UI, le même `ViewModel` que tu livres sur Android, les appels OS et le DSL de packaging forment un seul langage et un seul call graph. Pour les équipes déjà en Kotlin, la séparation frontend/backend disparaît.

## Rendu

Tauri rend via la WebView OS : WebView2 sur Windows, WKWebView sur macOS, WebKitGTK sur Linux. Cela garde les binaires petits et profite d'une accélération matérielle déjà calibrée par l'OS, au prix de différences de rendu selon l'OS, du CSS pour les widgets natifs et d'une fluidité d'animation liée au compositor de la WebView.

Nucleus rend l'UI Compose vers Skia via Skiko — Metal sur macOS, OpenGL ou Direct3D ailleurs — donc le renderer est le même sur chaque OS. Les gestes stylet et trackpad arrivent sous forme d'événements typés, et le [backend Tao](/docs/tao) ajoute le support Wayland natif sur Linux.

## Décorations de fenêtre natives

Le chrome de fenêtre de Tauri est ce que la WebView OS rend ; les title bars personnalisées exigent du CSS qui approche le comportement natif.

Nucleus fournit quatre thèmes [toolkit](/docs/window/toolkits) pour les fenêtres décorées — Liquid Glass (macOS 26+), Fluent (Windows), Yaru (Ubuntu) et Jewel (IntelliJ Platform). Chacun se rend en Compose, s'appuie sur la géométrie réelle d'une title bar native et se thématise depuis ton design system.

## Là où Tauri convient

- Taille de binaire : 5–20 MB, plus petit que Nucleus à 38 MB (GraalVM) ou 120 MB (JVM+AOT).
- Les équipes JavaScript-first qui veulent garder un frontend web avec un backend Rust.
- L'accès à l'écosystème de crates Rust côté backend. Nucleus s'appuie plutôt sur l'écosystème JVM.

## Là où Nucleus convient

- Un seul langage de bout en bout : UI, logique métier, appels OS et packaging en Kotlin.
- Un rendu natif avec Compose et Skia et des toolkits de décoration de fenêtre par OS, sans WebView.
- Multi-touch, pression et tilt du stylet, et Wayland natif sur Linux via le backend Tao.
- L'écosystème de bibliothèques JVM in-process — Lucene, DuckDB, ONNX Runtime, Apache Tika — ajouté comme dépendance `implementation`.
- 16 formats de packaging contre 5 — voir la [comparaison packaging](/docs/compare/packaging).

## Quand choisir Tauri

Choisis Tauri quand tu veux le plus petit binaire possible, que répartir le travail entre JavaScript et Rust te convient, et que ton UI a une forme web qui s'accommode bien de la WebView.

## Quand choisir Nucleus

Choisis Nucleus quand tu veux un seul langage de bout en bout, un rendu natif, le support multi-touch, stylet et Wayland, et l'écosystème de bibliothèques JVM in-process — surtout si tu livres déjà du Kotlin sur Android, iOS ou le web et que tu veux que le desktop s'aligne.
