`linux-hidpi` détecte le facteur d'échelle natif de l'affichage sous Linux et l'applique au
pipeline de rendu AWT. L'OpenJDK standard ignore le facteur d'échelle du bureau, si bien
qu'une fenêtre Compose Desktop s'affiche en 1x sur un écran HiDPI. Le module reprend la
logique de détection de JetBrains Runtime (`systemScale.c`) dans un bridge JNI, pour que
n'importe quelle JVM — y compris une image native GraalVM — résolve la bonne échelle.

<Callout type="info">
JetBrains Runtime gère le scaling par moniteur tout seul. `linux-hidpi` s'adresse à l'OpenJDK
standard et aux images natives GraalVM, où JBR n'est pas disponible.
</Callout>

## Ajouter la dépendance

```kotlin title="build.gradle.kts"
dependencies {
    implementation("dev.nucleusframework:nucleus.linux-hidpi:<version>")
}
```

Tu peux omettre cette dépendance si tu dépends déjà de
[`graalvm-runtime`](/docs/performance/graalvm/runtime-bootstrap) : il tire `linux-hidpi`
transitivement, et `GraalVmInitializer` appelle `applyLinuxHiDpiScale()` pour toi.

## Appliquer le facteur d'échelle

Appelle `applyLinuxHiDpiScale()` avant l'initialisation d'AWT, c'est-à-dire avant
`application {}` :

```kotlin
import androidx.compose.ui.window.application
import dev.nucleusframework.hidpi.applyLinuxHiDpiScale

fun main() {
    applyLinuxHiDpiScale()
    application {
        // ton app Compose Desktop
    }
}
```

La fonction est un no-op sur les plateformes non-Linux, tu peux donc laisser l'appel en place
sous macOS et Windows.

Pour lire la valeur toi-même — par exemple pour la journaliser ou la passer à un autre
sous-système — appelle `getLinuxNativeScaleFactor()` et définis la propriété manuellement :

```kotlin
import androidx.compose.ui.window.application
import dev.nucleusframework.hidpi.getLinuxNativeScaleFactor

fun main() {
    if (System.getProperty("sun.java2d.uiScale") == null) {
        val scale = getLinuxNativeScaleFactor()
        if (scale > 0.0) {
            System.setProperty("sun.java2d.uiScale", scale.toString())
        }
    }

    application {
        // ton app Compose Desktop
    }
}
```

## Fonctionnement

`getLinuxNativeScaleFactor()` consulte six sources de détection par ordre de priorité et
renvoie la première valeur positive trouvée :

| Priorité | Source | Description |
|----------|--------|-------------|
| 1 | `J2D_UISCALE` | Override JVM explicite (variable d'env) |
| 2 | GSettings | GNOME `org.gnome.desktop.interface` → `scaling-factor` |
| 3 | Mutter DBus | Échelle fractionnaire GNOME via `org.gnome.Mutter.DisplayConfig` |
| 4 | `GDK_SCALE` | Variable de session GTK / GNOME |
| 5 | `GDK_DPI_SCALE` | Multiplicateur DPI fractionnaire GTK |
| 6 | `Xft.dpi` | X Resource Manager (KDE, GNOME legacy, …) |

La détection s'exécute dans du code natif chargé via l'objet JNI `HiDpiLinuxBridge`. Si la
bibliothèque native échoue à se charger — par exemple dans un conteneur musl sans glibc — la
fonction retombe sur un parsing en Kotlin pur des variables d'environnement `J2D_UISCALE`,
`GDK_SCALE` et `GDK_DPI_SCALE`. Quand aucune source ne fournit de valeur, elle renvoie `0.0`
et laisse la JVM décider.

`applyLinuxHiDpiScale()` s'exécute lorsque l'échelle détectée est positive et que
`sun.java2d.uiScale` n'est pas déjà définie. Elle définit d'abord `GDK_SCALE` dans
l'environnement du processus via le bridge natif, ce qui fait passer le rendu et la mise à
l'échelle des coordonnées de souris par le chemin de détection `X11GraphicsDevice` du JDK — le
même que celui de JBR. En repli, elle définit aussi `sun.java2d.uiScale.enabled` et
`sun.java2d.uiScale`, pour que le rendu reste mis à l'échelle sur les JDK qui ne lisent pas
`GDK_SCALE`.

<Callout type="info">
AWT arrondit à l'échelle entière la plus proche, donc le scaling fractionnaire de GNOME
(125%, 150%) reste approximatif. Pour un scaling par moniteur et fractionnaire sous Linux,
utilise le [backend Tao](/docs/tao/hidpi), qui parle directement au compositeur.
</Callout>

## Référence de l'API

| API | Description |
|-----|-------------|
| `fun getLinuxNativeScaleFactor(): Double` | Facteur d'échelle résolu (par exemple `1.0`, `1.25`, `2.0`). Renvoie `0.0` si la détection échoue ou si la plateforme n'est pas Linux. |
| `fun applyLinuxHiDpiScale()` | Définit `sun.java2d.uiScale` (et `GDK_SCALE` via le bridge natif) quand la propriété n'est pas définie et que l'échelle détectée est supérieure à `0`. No-op sur les plateformes non-Linux. |

Les deux fonctions se trouvent dans le package `dev.nucleusframework.hidpi`.

## Notes

### Binaires natifs

Le module embarque des bibliothèques partagées pré-compilées dans le JAR pour Linux x64 et
aarch64, toutes deux nommées `libnucleus_linux_hidpi_jni.so`. `NativeLibraryLoader` de
`core-runtime` extrait le binaire correspondant à la première utilisation.

### ProGuard

Le plugin Gradle Nucleus ajoute cette règle automatiquement :

```proguard
-keep class dev.nucleusframework.hidpi.HiDpiLinuxBridge {
    native <methods>;
}
```
