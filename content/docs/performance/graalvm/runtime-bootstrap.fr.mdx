`nucleus.graalvm-runtime` prépare une app Compose Desktop à s'exécuter sous un build GraalVM
native-image. Un seul appel à `GraalVmInitializer.initialize()` depuis `main()` définit les
propriétés JVM, précharge le gestionnaire de polices et installe les substitutions de plateforme
dont native-image a besoin pour atteindre le premier frame.

## Ajouter la dépendance

```kotlin title="build.gradle.kts"
dependencies {
    implementation("dev.nucleusframework:nucleus.graalvm-runtime:<version>")
}
```

Si tu dépends de `nucleus.nucleus-application`, ce module est tiré transitivement et
`GraalVmInitializer.initialize()` s'exécute dans `nucleusApplication { … }`. Tu n'as pas besoin
de l'appeler toi-même.

## Initialiser le runtime

Appelle `initialize()` comme première instruction de `main()`, avant tout code AWT ou Compose :

```kotlin
import dev.nucleusframework.graalvm.GraalVmInitializer
import androidx.compose.ui.window.Window
import androidx.compose.ui.window.application

fun main() {
    GraalVmInitializer.initialize()

    application {
        Window(onCloseRequest = ::exitApplication, title = "MyApp") {
            App()
        }
    }
}
```

L'appel est sans danger sur une JVM classique. Tout ce qui est spécifique à native-image est
conditionné par la propriété système `org.graalvm.nativeimage.imagecode` ; sur une JVM standard,
seule la détection d'échelle HiDPI Linux s'exécute, et c'est un no-op sous macOS et Windows.

Pour sauter du code qui n'a de sens que sur la JVM, branche sur
`GraalVmInitializer.isNativeImage`.

## Fonctionnement

Sous une native image, `initialize()` met en place l'environnement qu'un lanceur préparerait
normalement avant `main()` :

| Aspect | Action |
|--------|--------|
| Look and feel Swing | Met `swing.defaultlaf` à Metal pour éviter les modules de look-and-feel natifs que native-image ne livre pas. |
| `java.home` | Pointe vers le répertoire de l'exécutable pour que Skiko trouve `jawt`. |
| `java.library.path` | Met `execDir` et `execDir/bin` pour que `fontmanager`, `freetype` et `awt` soient trouvables, puis vide le cache de chemins du `ClassLoader`. |
| Init des charsets | Appelle `Charset.defaultCharset()` tôt pour éviter `InternalError: platform encoding not initialized`. |
| Locale macOS | Lit la langue d'interface préférée depuis CoreFoundation et l'applique à `Locale.getDefault()`, car SubstrateVM reste sinon sur la locale POSIX `C`. |
| Préchargement du gestionnaire de polices | Appelle `System.loadLibrary("fontmanager")` pour éviter les crashs dans `Font.createFont()`. |
| HiDPI Linux | Détecte et applique le facteur d'échelle natif via [linux-hidpi](/docs/performance/linux-hidpi). Cette étape s'exécute sur chaque JVM, pas seulement les native images. |

native-image exécute le code Java deux fois : une fois au build, pendant la construction de
l'image, puis à nouveau au runtime, au démarrage du binaire. Les initializers statiques
s'exécutent au build sauf s'ils sont explicitement différés. Nucleus organise ses modules
runtime pour que les données pures (catalogues d'icônes, registres d'interfaces sealed, schémas
JSON) s'initialisent au build, sans coût au démarrage, tandis que tout ce qui touche l'OS
(système de fichiers, APIs OS, bibliothèques natives) est différé au runtime via la metadata
`RuntimeClassInitialization` ou l'initialisation paresseuse d'un `object`.

Si tu as un initializer statique qui touche l'OS, déclare-le sous `RuntimeClassInitialization`
dans ton `nativeImageConfigBaseDir`. Le Oracle Reachability Repository couvre déjà les cas
courants comme SLF4J et Logback.

## Substitutions SVM

Le module livre des substitutions `@TargetClass` qui patchent les internes d'AWT sous
native-image. Le compilateur native-image les prend en compte automatiquement, sans flag DSL :

- `FontCreateFontSubstitution` — bufferise `Font.createFont(int, InputStream)` vers un fichier
  temporaire sous Windows et Linux, pour contourner les streams sans support de mark/reset.
- `Win32FontManagerSubstitution` — une réimplémentation pure-Java de
  `Win32FontManager.getFontPath()` sous Windows, qui évite
  `InternalError: platform encoding not initialized`.
- `FcFontManagerSubstitution` — l'équivalent Fontconfig sous Linux, avec un repli sur
  `/usr/share/fonts`.
- `SplashScreenSubstitution` — fait renvoyer `null` à `java.awt.SplashScreen.getSplashScreen()`,
  car le splash screen est une fonction du lanceur sans backing JNI sous native-image.
- `XToolkitAppNameSubstitution` — remplace `XToolkit.getAWTAppClassName()` sous Linux pour que le
  `WM_CLASS` X11 reflète le vrai nom de l'application plutôt qu'un frame interne de SubstrateVM.

## Inclusion automatique des ressources

Un fichier `native-image.properties` livré avec le JAR enregistre quatre globs de ressources au
build, aux côtés des options `-H:+UnlockExperimentalVMOptions -H:+SharedArenaSupport` :

| Pattern | Couvre |
|---------|--------|
| `.*\.(svg\|ttf\|otf)` | Icônes SVG et polices du classpath : Jewel, IntelliJ Platform, ressources Compose, et les tiennes |
| `composeResources/.*` | Ressources Compose Multiplatform chargées via `Res.*` |
| `nucleus/.*` | Ressources natives Nucleus, y compris les bibliothèques partagées JNI |
| `META-INF/services/.*` | Descripteurs `ServiceLoader`, comme les moteurs Ktor, les fetchers Coil et les providers SLF4J |

Grâce à cela, les icônes ne s'affichent pas comme des carrés vides sous native-image.

<Callout type="info" title="Compromis sur la taille du binaire">
Le glob `.*\.(svg|ttf|otf)` est large à dessein. Si tu livres la bibliothèque d'icônes IntelliJ
Platform et que tu tiens à chaque mégaoctet, remplace-le par un `resource-config.json` plus
étroit.
</Callout>

## Notes

Utilise le module [decorated-window-jni](/docs/window) sous native-image. Il ne dépend pas de
JetBrains Runtime. Réserve `decorated-window-jbr` aux builds JVM.
