L'exemple [`benchmark-demo`](https://github.com/NucleusFramework/Nucleus/tree/main/examples/benchmark-demo)
exécute une suite de style Geekbench — 13 kernels CPU, des montées en charge de rendu géométriques
et un bench de chargement de liste — à l'intérieur d'une vraie application desktop Compose, puis
réimplémente les mêmes kernels à l'identique en SwiftUI, Tauri et Flutter. Chaque variante exécute
la même suite sur la même machine, donc les chiffres sont directement comparables. Cette page
présente les résultats mesurés.

## Contexte

- **Date :** 2026-07-16
- **Machine :** Apple M4, 10 cœurs, sur secteur, thermiquement stable
- **Toolchains :** Oracle GraalVM 25.1.3+9.1 (Java 25.0.3), OpenJDK 26.0.1, daemon Gradle JBR 21
- **Protocole :** chaque variante exécute elle-même sa suite (3 warmups écartés, puis best-of-5 sur les kernels CPU). Les exécutions sont strictement séquentielles, une application à la fois ; le pic de RSS est échantillonné pendant chaque exécution. Les self-checks intégrés passent partout.

<Callout type="info">
Les scores composites ne sont comparables qu'au sein d'une suite identique. Chaque valeur ci-dessous
provient de la même suite et de la même session machine. CPU et GFX : plus c'est haut, mieux c'est.
list, pic de RAM et taille de l'app : plus c'est bas, mieux c'est.
</Callout>

## Points clés

- **CPU brut.** GraalVM JIT est en tête (146.7). Les baselines LLVM (Tauri 142, SwiftUI 139) et OpenJDK C2 (138–140) sont groupées. GraalVM Native ne rejoint le peloton JVM/LLVM qu'**avec le PGO** (100.8 → 132.7) ; sans profil, `matmul` s'effondre (3877 contre 12225 MFLOP/s). `-O2` et `-Os` restent nettement derrière.
- **Le sweet spot du PGO.** `aot-o3-pgo` atteint un CPU proche de la JVM (132.7) avec **2,5× moins de RAM** (269 Mo contre 616–695 Mo), un `.app` compact (129 Mo) et un démarrage rapide. Le coût : le flux PGO (build instrumenté → exécution GUI → rebuild). Le PGO réduit aussi l'image (174 Mo → 129 Mo).
- **RAM.** Flutter est le plus léger (196 Mo), devant SwiftUI (268) et GraalVM Native (269–326). Les JVM sont à 616–695 Mo. Tauri est le plus lourd (794 Mo) une fois l'arbre de processus WebKit compté (†).
- **Démarrage / liste.** Natif ~17 ms, JVM ~21 ms, SwiftUI 14 ms (le meilleur), Tauri 233 ms (rendu de liste DOM/canvas WebView — un ordre de grandeur plus lent).
- **CPU le plus bas : Flutter** (57.3) : `raytracer` au quart du peloton, `matmul`/`sha`/`sieve` faibles — mais compensé par un rendu de premier plan (GFX 15133).

<Callout type="warn" title="RAM de Tauri — le piège WebView multi-processus (†)">
Tauri/WKWebView est multi-processus sur macOS. Le canvas et le JS tournent dans des processus WebKit
séparés (`com.apple.WebKit.WebContent`, `.GPU`, `.Networking`), pas dans le binaire de l'app. Le RSS
du processus principal seul est de 177 Mo, mais l'empreinte réelle — en sommant l'arbre de processus
WebKit — est de 794 Mo, ce qui fait de Tauri la variante la plus lourde en RAM, pas la plus légère.
(Sommer les RSS surcompte légèrement les pages partagées de `WebKit.framework`, donc 794 Mo est une
borne haute ; l'écart 177→794 est large et réel.)
</Callout>

<Callout type="info" title="Taille de l'app — autonome vs runtime système (‡)">
Les variantes autonomes embarquent leur propre runtime, donc leur poids est le `.app` complet
(runtime + Skiko/dylibs + jars/binaire) : JVM (jlink), GraalVM Native, Flutter. SwiftUI et Tauri
s'appuient sur le runtime de l'OS — SwiftUI sur le SwiftUI/AppKit système, Tauri sur le WebKit
système — donc leurs binaires (0.34 / 5.6 Mo) excluent le moteur fourni par macOS. Les deux classes
de taille ne sont pas directement comparables.
</Callout>

## Reproduire

```bash title="terminal"
cd examples/benchmark-demo

# Matrice JVM + natif (RAM homogène) :
JAVA_HOME=<jbr-21> GRAALVM_HOME=<graalvm-25> JDK_C2_HOME=<jdk-26> \
  ONLY="jvm-c2 jvm-c2-pg jvm-graal aot-os aot-o2 aot-o3 aot-o3-pgo" ./run-all.sh

# ports :
ONLY="swiftui tauri flutter" ./run-all.sh
```

Les builds natifs doivent tourner avec `JAVA_HOME=<graalvm>` (la JVM courante doit être GraalVM).
Sinon, un JDK Oracle 25 provisionné par Gradle mais sans `native-image` peut correspondre au spec
`vendor=ORACLE`, `version=25` et être sélectionné, faisant échouer la compilation. Chaque variante
écrit un JSON de résultats ; l'annexe complète par variante est dans le `RESULTS.md` de l'exemple.
