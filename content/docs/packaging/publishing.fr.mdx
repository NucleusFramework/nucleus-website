Dans ce tutoriel, tu vas configurer un backend de publication pour que tes installateurs signés et
leurs métadonnées d'auto-update arrivent au même endroit — GitHub Releases, un bucket S3 ou
n'importe quel host HTTP. Tu choisiras un canal et un type de release, tu contrôleras le moment où
la publication s'exécute, et tu verras ce que chaque store desktop attend pour une soumission. La
configuration vit dans le même DSL `nucleus { }` que le reste de ton build.

## Avant de commencer

- Des installateurs fonctionnels pour tes plateformes cibles. Voir
  [Configuration](/docs/start/configuration) pour le réglage de `targetFormats(...)`.
- Des builds signés pour les plateformes que tu distribues. Voir
  [signature de code](/docs/packaging/code-signing).
- Une cible d'hébergement : un dépôt GitHub, un bucket S3 ou un serveur HTTP que tu contrôles.

## Configurer un backend de publication

Ajoute un bloc `publish { }` dans `nativeDistributions { }` et active un backend. Le bloc génère
seulement la configuration et le YAML d'update ; l'upload lui-même s'exécute en CI.

<Tabs items={['GitHub Releases', 'S3', 'HTTP générique']}>
<Tab value="GitHub Releases">

```kotlin title="build.gradle.kts"
nucleus {
    application {
        nativeDistributions {
            publish {
                github {
                    enabled = true
                    owner = "myorg"
                    repo = "myapp"
                    token = System.getenv("GITHUB_TOKEN")
                    channel = ReleaseChannel.Latest
                    releaseType = ReleaseType.Release
                }
            }
        }
    }
}
```

Une release se retrouve avec les installateurs et leur YAML d'auto-update côte à côte :

```
v1.0.0 (Release)
├── MyApp-1.0.0-macos-arm64.dmg
├── MyApp-1.0.0-macos-universal.dmg
├── MyApp-1.0.0-windows-amd64.exe
├── MyApp-1.0.0-windows.msixbundle
├── MyApp-1.0.0-linux-amd64.deb
├── MyApp-1.0.0-linux-amd64.AppImage
├── latest-mac.yml
├── latest.yml
└── latest-linux.yml
```

Le token a besoin de la permission `contents: write`. Sur GitHub Actions, mets
`permissions: contents: write` sur le job.

</Tab>
<Tab value="S3">

Utilise un bucket S3, éventuellement derrière un CDN, pour de la distribution auto-hébergée :

```kotlin title="build.gradle.kts"
publish {
    s3 {
        enabled = true
        bucket = "my-updates-bucket"
        region = "us-east-1"
        path = "releases/myapp"
        acl = "public-read"
    }
}
```

Fournis `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY` et `AWS_REGION` dans l'environnement.

</Tab>
<Tab value="HTTP générique">

N'importe quel host statique convient — Nginx, Caddy, Cloudflare R2, ou S3 en accès public. Le
plugin génère les métadonnées YAML pour ton URL de base ; tu uploades les fichiers toi-même :

```kotlin title="build.gradle.kts"
publish {
    generic {
        enabled = true
        url = "https://updates.example.com/releases/"
        channel = ReleaseChannel.Latest
        useMultipleRangeRequest = true
    }
}
```

`useMultipleRangeRequest` active les téléchargements différentiels et ne fonctionne que si ton
serveur honore les requêtes HTTP `Range`.

</Tab>
</Tabs>

## Choisir un canal et un type de release

Le canal sélectionne le fichier YAML que lit l'auto-updater, ainsi que le pattern de tag associé :

| Canal | YAML | Pattern de tag |
|-------|------|----------------|
| `ReleaseChannel.Latest` | `latest-*.yml` | `v1.0.0` |
| `ReleaseChannel.Beta` | `beta-*.yml` | `v1.0.0-beta.1` |
| `ReleaseChannel.Alpha` | `alpha-*.yml` | `v1.0.0-alpha.1` |

Pour GitHub Releases, le type de release contrôle la visibilité :

| Type | Comportement |
|------|--------------|
| `ReleaseType.Release` | Public sur la page des releases |
| `ReleaseType.Draft` | Masqué jusqu'à publication manuelle |
| `ReleaseType.Prerelease` | Marqué comme pré-release |

## Définir le mode de publication

`publishMode` décide du moment où la publication s'exécute. Sa valeur par défaut est
`PublishMode.Never` :

```kotlin title="build.gradle.kts"
publish {
    publishMode = PublishMode.Auto
    // github { ... }
}
```

- `PublishMode.Never` — ne publie pas. Les métadonnées `latest-*.yml` sont tout de même générées
  localement pour chaque format à jour.
- `PublishMode.Auto` — publie quand le build part d'un tag git.
- `PublishMode.Always` — publie à chaque build, avec ou sans tag.

<Callout type="info">
Le bloc `publish { }` écrit les paramètres publisher d'electron-builder ; il n'uploade rien par
lui-même. L'upload réel s'exécute dans ton workflow CI (`gh release create` pour GitHub,
`aws s3 cp` pour S3, ta propre étape de copie pour le générique). Voir la
[CI de release](/docs/packaging/ci-cd#release) pour le chemin standard.
</Callout>

## Soumettre à un store desktop

La distribution en store passe par le portail de chaque éditeur plutôt que par le bloc
`publish { }`. Construis le format store, puis uploade l'artefact :

- **Mac App Store** — soumets le PKG via Transporter ou `xcrun altool`. Les provisioning profiles
  ([sandboxing](/docs/packaging/sandboxing)) et la catégorie du store (`appCategory`) sont fixés au
  build. Apple review le build ; les builds App Store ne sont pas notarisés.
- **Microsoft Store** — uploade le `.msixbundle` sur
  [Partner Center](https://partner.microsoft.com/). L'identité doit correspondre à la réservation
  (`identityName`, `publisher = CN=<GUID>`, `publisherDisplayName`). Voir
  [cibles Windows](/docs/packaging/windows#appx-msix).
- **Snapcraft** — lance `snapcraft upload` depuis la CI. Le Snap Store accepte les confinements
  `strict` et `classic` ; `classic` exige une approbation. Définis le grade avec
  `linux.snap { grade = SnapGrade.Stable }`.
- **Flathub** — soumets une pull request de manifest au
  [dépôt Flathub](https://github.com/flathub/flathub). Ton bloc `flatpak` produit le bundle ;
  Flathub build à partir des sources via ton manifest.
- **GitHub Releases / téléchargement direct** — le chemin par défaut. Combine-le avec
  l'[auto-update](/docs/packaging/auto-update) pour les apps auto-distribuées.

## Référence

- `publish { publishMode = … }`
- `publish.github { enabled, owner, repo, token, channel, releaseType }`
- `publish.s3 { enabled, bucket, region, path, acl }`
- `publish.generic { enabled, url, channel, useMultipleRangeRequest }`
- `enum class PublishMode { Never, Auto, Always }`
- `enum class ReleaseChannel { Latest, Beta, Alpha }`
- `enum class ReleaseType { Release, Draft, Prerelease }`

Voir la [référence Gradle DSL](/docs/reference/gradle-dsl) pour la surface complète.
